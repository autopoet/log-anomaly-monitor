# 分布式日志异常检测与监控系统设计报告

## 一、项目概述

本项目实现了一个基于 MOM 消息队列技术的分布式日志采集、异常分析与实时可视化监控系统。系统通过多个日志采集节点模拟不同设备的运行日志，并将日志消息发送到 RabbitMQ 队列中。日志分析服务订阅原始日志消息，按照 `device_id` 对不同设备进行独立统计和异常检测，最后由 Web 监控服务实时展示设备状态、WARN/ERROR 占比、最近一次 ERROR 事件以及严重告警信息。

项目主要用于完成课程第二次作业，同时尽量保持清晰的工程结构，便于运行、展示和后续扩展。

## 二、需求分析

### 1. 日志采集需求

系统需要支持多个日志采集节点。实验中使用多个 Python 进程模拟分布式采集节点，每个节点都有唯一的节点编号。采集节点默认每 100ms 生成一条日志消息，并发布到消息队列中。

日志消息采用 JSON 格式：

```json
{
  "device_id": "device_01",
  "timestamp": "2026-04-23 12:00:00",
  "log_level": "INFO",
  "message": "system is healthy"
}
```

其中 `log_level` 包含 `INFO`、`WARN`、`ERROR` 三种级别。

### 2. 日志分析需求

日志分析服务需要订阅所有原始日志消息，并针对不同设备分别维护分析状态。对每个 `device_id`，系统需要维护最近 `N` 条日志记录，并计算以下信息：

- ERROR 级别日志占比。
- WARN 级别日志占比。
- 最近一次 ERROR 事件。
- 最近一次 ERROR 事件的时间戳。

分析服务每隔 `T` 秒将分析结果打包成新消息，发布到分析结果队列。

### 3. 严重告警需求

系统需要在最近 `S` 秒内检查设备日志。如果某台设备在该时间窗口内 ERROR 占比超过 50%，则生成严重告警消息，并发布到告警队列。

为了避免同一设备在持续异常期间重复发送大量告警，系统只在设备从正常状态进入严重状态时产生新的告警。

### 4. 实时可视化需求

Web 监控服务需要实时展示不同设备的运行状态，主要包括：

- 各设备 WARN/ERROR 占比。
- 最近一次 ERROR 事件及时间。
- 严重告警状态和告警次数。
- 过去一段时间内 WARN/ERROR 数量变化趋势折线图。

前端页面通过 WebSocket 接收实时分析结果和告警消息，并使用 ECharts 绘制趋势图。

## 三、总体架构设计

系统整体由五部分组成：

- 日志采集节点 `collector`
- RabbitMQ 消息队列
- 日志分析服务 `analyzer`
- SQLite 数据库
- Web 监控服务 `web`

整体数据流如下：

```text
日志采集节点 -> RabbitMQ logs.raw -> 日志分析服务 -> RabbitMQ logs.analysis
                                      |            -> RabbitMQ logs.alerts
                                      v
                                   SQLite
                                      ^
                                      |
                               FastAPI + WebSocket -> 实时监控页面
```

这种结构将日志采集、日志分析和实时展示解耦。采集节点只负责生成和发送日志，分析服务只负责消费和统计日志，Web 服务只负责展示分析结果。各模块之间通过消息队列进行通信，符合 MOM 消息队列的设计思想。

## 四、消息队列设计

系统使用 RabbitMQ 作为消息中间件，设计了三个队列：

| 队列名 | 生产者 | 消费者 | 作用 |
| --- | --- | --- | --- |
| `logs.raw` | 日志采集节点 | 日志分析服务 | 传递原始日志消息 |
| `logs.analysis` | 日志分析服务 | Web 监控服务 | 传递周期性分析结果 |
| `logs.alerts` | 日志分析服务 | Web 监控服务 | 传递严重告警消息 |

通过队列划分，可以让不同类型的消息职责更加清晰，也方便后续扩展其他消费者，例如邮件告警服务、日志归档服务等。

## 五、模块设计

### 1. `collector` 模块

`collector` 模块用于模拟分布式日志采集节点。核心文件为 `collector/producer.py`。

该模块主要功能包括：

- 支持通过 `--node-id` 指定采集节点编号。
- 随机选择模拟设备生成日志。
- 按不同设备配置不同 INFO/WARN/ERROR 生成概率。
- 将日志消息发布到 RabbitMQ 的 `logs.raw` 队列。

项目中预设了几类设备，例如正常设备、WARN 较多的设备、ERROR 较多的设备和状态波动设备，便于演示异常检测效果。

### 2. `analyzer` 模块

`analyzer` 模块用于日志消费、统计分析和严重告警判断。主要文件包括：

- `analyzer/consumer.py`
- `analyzer/detector.py`

其中 `consumer.py` 负责连接 RabbitMQ、消费原始日志、发布分析结果和告警消息；`detector.py` 负责核心异常检测逻辑。

分析服务对每台设备维护独立状态，不同设备之间互不影响。

### 3. `web` 模块

`web` 模块使用 FastAPI 实现，主要文件包括：

- `web/main.py`
- `web/preview.py`
- `web/static/index.html`
- `web/static/app.js`
- `web/static/style.css`

`web/main.py` 是完整实验模式的 Web 服务入口，负责读取 SQLite 数据，并消费 `logs.analysis` 和 `logs.alerts` 队列，将消息通过 WebSocket 推送给前端。

`web/preview.py` 是可视化预览模式入口，在没有 Docker 或 RabbitMQ 的环境下，可以生成模拟数据用于查看页面效果。该模式不替代完整实验链路。

### 4. `common` 模块

`common` 模块保存公共代码：

- `config.py`：读取环境变量配置。
- `models.py`：定义日志、分析结果和告警的数据模型。
- `mq.py`：封装 RabbitMQ 队列声明、发布和消费。
- `storage.py`：封装 SQLite 建表、写入和查询。

公共模块减少了重复代码，也让采集、分析和 Web 服务之间的数据结构保持一致。

### 2. 分析结果模型 `AnalysisResult`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `device_id` | 字符串 | 设备编号 |
| `timestamp` | 时间 | 分析结果生成时间 |
| `total_count` | 整数 | 当前窗口日志总数 |
| `warn_count` | 整数 | WARN 数量 |
| `error_count` | 整数 | ERROR 数量 |
| `warn_ratio` | 小数 | WARN 占比 |
| `error_ratio` | 小数 | ERROR 占比 |
| `latest_error_message` | 字符串或空 | 最近一次 ERROR 内容 |
| `latest_error_timestamp` | 时间或空 | 最近一次 ERROR 时间 |

| `alert_count` | 整数 | 累计告警次数 |

| 字段 | 类型 | 说明 |

| `error_ratio` | 小数 | 告警窗口内 ERROR 占比 |
| `window_seconds` | 整数 | 检测时间窗口 |
| `message` | 字符串 | 告警说明 |

## 七、异常检测算法设计

系统对每台设备维护两个窗口：

### 1. 最近 N 条日志窗口

该窗口用于周期性统计 WARN/ERROR 占比。每当收到一条新日志，就将其加入设备对应的队列中。如果队列长度超过 `N`，则移除最早的一条日志。

统计公式如下：

```text
WARN 占比 = WARN 数量 / 最近 N 条日志总数
ERROR 占比 = ERROR 数量 / 最近 N 条日志总数
```

### 2. 最近 S 秒时间窗口

该窗口用于严重告警判断。系统从当前设备的最近日志中筛选出时间戳位于最近 `S` 秒内的日志，计算其中 ERROR 占比。

告警条件如下：

```text
最近 S 秒内 ERROR 数量 / 最近 S 秒内日志总数 > 0.5
```

当条件满足且设备之前不处于严重状态时，系统生成一条严重告警消息。如果设备已经处于严重状态，则不重复发送告警。

## 八、数据持久化设计

项目使用 SQLite 保存运行数据。主要包含三张表：

- `logs`：保存原始日志。
- `analysis_results`：保存周期性分析结果。
- `alerts`：保存严重告警记录。

SQLite 的优点是轻量、无需单独安装数据库服务，适合课程实验和本地演示。Web 页面刷新后，也可以通过 SQLite 查询最近的分析结果和告警记录。

## 九、接口设计

Web 服务提供以下接口：

| 接口 | 说明 |
| --- | --- |
| `GET /` | 打开实时监控页面 |
| `GET /api/devices` | 查询所有设备最新状态 |
| `GET /api/devices/{device_id}/summary` | 查询指定设备最新状态 |
| `GET /api/devices/{device_id}/trend` | 查询指定设备趋势数据 |
| `GET /api/alerts` | 查询最近严重告警 |
| `WS /ws/monitor` | 实时推送分析结果和告警消息 |

前端页面首次加载时会请求历史数据，随后通过 WebSocket 接收实时更新。

## 十、运行方式

### 1. Docker 一键启动完整系统

如果环境中安装了 Docker，可以执行：

```bash
docker compose up --build
```

该方式会启动 RabbitMQ、日志采集节点、日志分析服务和 Web 监控服务。

### 2. 手动启动完整实验链路

也可以先启动 RabbitMQ，再分别启动分析服务、采集服务和 Web 服务：

```bash
docker compose up -d rabbitmq
python -m analyzer.consumer
python -m collector.producer --node-id collector-01
python -m uvicorn web.main:app --reload
```

### 3. 可视化预览模式

如果没有 Docker 或 RabbitMQ，可以使用：

```bash
python -m uvicorn web.preview:app --reload
```

该模式只用于查看页面效果，不替代完整实验链路。

## 十二、总结

本项目完成了课程要求中的日志采集、消息队列传输、日志分析、异常检测和实时可视化展示功能。系统使用 RabbitMQ 实现模块间解耦，使用滑动窗口方法实现按设备维度的日志统计和严重告警判断，并通过 FastAPI、WebSocket 和 ECharts 实现实时监控页面。

项目整体结构较清晰，核心逻辑可以单独测试，同时也提供了 Docker 一键启动和无 RabbitMQ 的可视化预览模式，便于演示和提交。
